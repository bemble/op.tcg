# One Piece Collection

A small, self-hostable app to track a **One Piece TCG** card collection shared
between several people. Card data comes straight from the official card list
([en.onepiece-cardgame.com](https://en.onepiece-cardgame.com/cardlist)) — no API
key, no quota. Built to drop onto a homelab as a single Docker container.

What it does (deliberately minimal):

- **Collection by set** (home page): browse every set with a completion bar (owned / total), drill into a set to see all its cards — owned in colour, missing greyed (toggle to hide them) — and add/manage ownership from each card.
- Search One Piece cards by name and **add** them to the collection.
- For each card, record **who owns it** (one or several co-owners), the
  **language**, the **quantity**, and a free **comment**. A card can be owned
  several times, by one or several people.
- What matters first is **whether a card is owned or not** — the owner is
  optional detail you can fill in per card.
- Co-owners are managed as a free-text list in **Préférences**, then picked from
  a dropdown per card.

No multi-game, no regions, no accounts.

## Stack

- **Backend** — Go HTTP API. Two storage layers (see below): a JSON catalogue file + a small SQLite file for the collection (pure-Go `modernc.org/sqlite`, no CGO → static binary).
- **Frontend** — TypeScript SPA built with Vite, UI from [WebAwesome](https://webawesome.com/) web components, **bundled** (no CDN) so it works offline.
- **Storage** —
  - **Catalogue** → [`catalogue.json`](catalogue.json), a committed file holding the whole One Piece card catalogue (~4500 cards incl. parallels, all sets up to the latest). Loaded into memory at startup; search filters it in-memory. **Committed to the repo and shippable**, so the app works out of the box.
  - **Collection** → one SQLite file (owners + possessions) on a Docker volume.
- **No key, no quota** — `catalogue.json` is (re)generated by a *sync* that scrapes the official card list (~53 requests, one per set). It runs **automatically on startup** when the catalogue is empty or stale (`SYNC_ON_START` / `SYNC_MAX_AGE_HOURS`), in the background — the existing catalogue is served meanwhile and the UI shows a *"Synchronisation de la base de données…"* banner — and can be triggered manually from *Préférences*. **Search reads only the in-memory catalogue**: zero network calls, instant, offline.
- **Offline** — browsing *and searching* work without internet. Card *images* are fetched and downscaled server-side (the official host blocks cross-origin `<img>` loads) and optionally cached on disk; they render whenever the backend is online.
- **Auth** — none. Single LAN deployment; put it behind your own reverse proxy if you want a login. The SPA uses clean-URL (History API) routing, so serve it at the **site root** (or a subdomain) — not under a sub-path.

## Quick start (Docker)

```bash
docker compose up -d --build
```

Open http://localhost:8080. No API key, no `.env` needed.

Your collection lives in the `onepiece-data` volume. The bundled `catalogue.json`
seeds the catalogue on first run, so search works immediately. Hit **Synchroniser
le catalogue** in *Préférences* to refresh it from the official site when new
sets release.

## Configuration

All optional, via environment variables:

| Var               | Default                | Purpose                                       |
| ----------------- | ---------------------- | --------------------------------------------- |
| `PORT`            | `8080`                 | HTTP port.                                     |
| `DB_PATH`         | `./data/onepiece.db`   | SQLite file path (collection). Docker image sets `/data/onepiece.db`. |
| `CATALOGUE_PATH`  | `./catalogue.json`     | JSON catalogue file (read at startup, rewritten on sync). Docker: `/data/catalogue.json`. |
| `CATALOGUE_SEED`  | _(empty)_              | Optional bundled JSON used to seed `CATALOGUE_PATH` if missing. |
| `SYNC_ON_START`   | `true`                 | Refresh the catalogue from the official site on startup (background). |
| `SYNC_MAX_AGE_HOURS` | `24`                | Skip the startup sync if the catalogue is younger than this (0 = always). |
| `WEB_DIR`         | `./web`                | Directory of the built SPA the server serves (Docker: `/app/web`). |
| `IMAGE_WIDTH`     | `220`                  | Thumbnail width (px) for server-side image downscaling; 0 disables. |
| `IMAGE_CACHE`     | `false`                | Persist processed images to disk (`IMAGE_CACHE_DIR`, default `./data/imgcache`). |

## HTTP API

| Method   | Path                          | Description                                                  |
| -------- | ----------------------------- | ------------------------------------------------------------ |
| `GET`    | `/api/health`                 | Liveness + cached card count.                                |
| `GET`    | `/api/search?name=&page=&limit=` | Search the **in-memory** catalogue (name or code). No network call. |
| `GET`    | `/api/img?u=`                 | Server-side image proxy: fetches, downscales and re-serves a card image. |
| `GET`    | `/api/sync`                   | Sync status: catalogue card count, last sync time, in-progress flag, last error. |
| `POST`   | `/api/sync`                   | Start a background scrape of the official card list (~53 requests) and rewrite `catalogue.json`. Returns `202`; poll `GET /api/sync`. |
| `GET`    | `/api/cards/{id}`             | A catalogue card.                                           |
| `GET`    | `/api/sets`                   | Every set with completion (`owned`/`total` printed codes), in display order. |
| `GET`    | `/api/sets/{code}`            | All cards of a set (one per printed code) annotated with ownership. |
| `GET`    | `/api/collection?q=&owner=`   | The collection (from SQLite, offline). Filter by name / owner.|
| `POST`   | `/api/collection`             | Add a card. Body: `cardId`, `ownerId?`, `language`, `quantity`, `notes`. |
| `PATCH`  | `/api/collection/{id}`        | Update owner/language/quantity/notes (qty ≤ 0 deletes). Omitted fields are kept; `ownerId:null` clears the owner. |
| `DELETE` | `/api/collection/{id}`        | Remove a possession.                                         |
| `GET`    | `/api/collection/stats`       | Totals overall and per owner.                                |
| `GET`    | `/api/owners`                 | List co-owners.                                              |
| `POST`   | `/api/owners`                 | Add a co-owner. Body: `{ "name": "…" }`.                     |
| `DELETE` | `/api/owners/{id}`            | Remove a co-owner (their cards become *Non attribué*, not deleted). |

## Data model

**Catalogue** — `catalogue.json`: `{ syncedAt, count, cards: [...] }`, where each card is normalized (id, name, code, rarity, set, image) plus a `raw` object with the extra gameplay fields scraped from the official card list (color, type, power, counter, attribute, cost/life, effect, …). Loaded into memory at startup; the sync timestamp travels with the file.

**Collection** (SQLite, [`backend/schema.sql`](backend/schema.sql)):
- `owners` — co-owners (free-text names) managed in Préférences.
- `collection_items` — one row per **possession**: `(card_id, owner_id?, language, quantity, notes)`. `card_id` references a card in `catalogue.json` (resolved at the app layer; absent cards still render as a stub).
  Adding the same `(card_id, owner_id, language)` again increments the quantity.
  Deleting an owner sets `owner_id` to NULL (`ON DELETE SET NULL`) so no card is lost.

## Local development

Backend (needs Go 1.22+):

The backend reads a `.env` file automatically (looked up as `./.env` then
`../.env`, or via `ENV_FILE=`). **Run it from the repo root** so the local
defaults line up — `.env`, `data/` (the dev SQLite db) and `catalogue.json` all
sit at the repo root, next to each other:

```bash
(cd backend && go mod tidy)   # one-time
go run ./backend              # db -> ./data/, catalogue -> ./catalogue.json
```

Real environment variables still override `.env`, e.g. `PORT=9000 go run ./backend`.

Frontend (needs Node 20+), with hot reload and `/api` proxied to the backend:

```bash
cd frontend
npm install
npm run dev   # http://localhost:5173
```

`npm run build` type-checks and outputs the static SPA to `frontend/dist`, which the Go server serves in production.
