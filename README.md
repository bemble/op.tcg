# One Piece Collection

A small, self-hostable app to track a **One Piece TCG** card collection shared
between several people. Card data comes straight from the official card list
([en.onepiece-cardgame.com](https://en.onepiece-cardgame.com/cardlist)) — no API
key, no quota. Built to drop onto a homelab as a single Docker container.

What it does (deliberately minimal):

- **Collection by set** (home page): browse every set with a completion bar, drill into a set to see all its cards — owned cards show the flag(s) of the languages you have + a quantity, missing ones are greyed — and tap any card to add/manage it. Grid or list view; filter by status; sort parallels to the end.
- **Card statuses**: each copy is **owned** (a physical card), **ordered** (bought, not yet in hand — counts toward completion) or **wishlist** (wanted — doesn't count). Statuses are tracked **per owner**, so Milo can own a card while Pierre wishlists it.
- **Suivi tab**: a single place listing everything **ordered** and **wishlisted**, with quick edit/delete and image zoom, to find and update them fast.
- **Statistiques tab**: completion per goal, per family, per owner, per language, per rarity.
- Search One Piece cards by name/code and **add** them to the collection.
- For each card, record **who owns it** (one or several co-owners), the **language**, the **quantity**, a free **comment**, and its **status**. A card can be owned several times, by one or several people.
- **Bulk-edit** the language of many copies at once (list view).
- **Collection goal** (complete / master / wizard) and which **set families** to include, set in **Préférences** — they scope the completion counters.
- **Add a missing card** by hand from a TCGplayer product URL (e.g. promo alt-arts the official list omits), in **Préférences**.
- Co-owners are managed as a free-text list in **Préférences**, then picked from a dropdown per card.
- **Installable** as a PWA (iOS/Android home-screen app, standalone, offline-capable shell).
- **MCP server** at `/mcp` so an assistant can search, check completion and manage the collection (see below).

No multi-game, no regions, no accounts.

## Stack

- **Backend** — Go HTTP API. Two storage layers (see below): a JSON catalogue file + a small SQLite file for the collection (pure-Go `modernc.org/sqlite`, no CGO → static binary).
- **Frontend** — TypeScript SPA built with Vite, UI from [WebAwesome](https://webawesome.com/) web components, **bundled** (no CDN) so it works offline.
- **Storage** —
  - **Catalogue** → `catalogue.json`, holding the whole One Piece card catalogue (~4800 cards incl. parallels, all sets up to the latest). Loaded into memory at startup; search filters it in-memory. **Not committed** — each deployment builds it on first run via the startup background sync (see below), then keeps it on the data volume. (`CATALOGUE_SEED` can point at a bundled JSON to seed it offline if you have one.)
  - **Collection** → one SQLite file (owners + possessions + user-added "curated" cards) on a Docker volume.
- **No key, no quota** — `catalogue.json` is (re)generated by a *sync* that scrapes the official card list (~53 requests, one per set). It runs **automatically on startup** when the catalogue is empty or stale (`SYNC_ON_START` / `SYNC_MAX_AGE_HOURS`), in the background — the existing catalogue is served meanwhile and the UI shows a *"Synchronisation de la base de données…"* banner — and can be triggered manually from *Préférences*. **Search reads only the in-memory catalogue**: zero network calls, instant, offline.
- **Offline** — browsing *and searching* work without internet. Card *images* are fetched and downscaled server-side (the official host blocks cross-origin `<img>` loads) and optionally cached on disk; they render whenever the backend is online.
- **Auth** — none. Single LAN deployment; put it behind your own reverse proxy if you want a login. The SPA uses clean-URL (History API) routing, so serve it at the **site root** (or a subdomain) — not under a sub-path.

## Quick start (Docker)

```bash
docker compose up -d --build
```

Open http://localhost:8080. No API key, no `.env` needed.

Your collection lives in the `onepiece-data` volume. On **first run** the app has
no catalogue yet, so it runs a background sync against the official card list to
build `catalogue.json` (needs internet once; the UI shows a syncing banner
meanwhile). Afterwards it's served from memory and works offline. Hit
**Synchroniser le catalogue** in *Préférences* to refresh it when new sets release.

## Configuration

All optional, via environment variables:

| Var               | Default                | Purpose                                       |
| ----------------- | ---------------------- | --------------------------------------------- |
| `PORT`            | `8080`                 | HTTP port.                                     |
| `DB_PATH`         | `./data/onepiece.db`   | SQLite file path (collection). Docker image sets `/data/onepiece.db`. |
| `CATALOGUE_PATH`  | `./catalogue.json`     | JSON catalogue file (read at startup, rewritten on sync). Docker: `/data/catalogue.json`. |
| `CATALOGUE_SEED`  | _(empty)_              | Optional bundled JSON used to seed `CATALOGUE_PATH` if missing. |
| `SYNC_ON_START`   | `true`                 | Refresh the catalogue from the official site on startup (background). |
| `SYNC_MAX_AGE_HOURS` | `24`                | Skip the startup sync if the catalogue is younger than this (0 = always). |
| `WEB_DIR`         | `./web`                | Directory of the built SPA the server serves (Docker: `/app/web`). |
| `IMAGE_WIDTH`     | `220`                  | Thumbnail width (px) for server-side image downscaling; 0 disables. |
| `IMAGE_CACHE`     | `false`                | Persist processed images to disk (`IMAGE_CACHE_DIR`, default `./data/imgcache`). |
| `MCP_ENABLED`     | `true`                 | Expose the MCP server at `/mcp`. |
| `MCP_READ_ONLY`   | `false`                | Register only the read tools (no add/update/remove). |
| `MCP_TOKEN`       | _(empty)_              | If set, `/mcp` requires `Authorization: Bearer <token>`. **Set this** if the app is reachable beyond your trusted network. |

## HTTP API

| Method   | Path                          | Description                                                  |
| -------- | ----------------------------- | ------------------------------------------------------------ |
| `GET`    | `/api/health`                 | Liveness + cached card count.                                |
| `GET`    | `/api/search?name=&page=&limit=` | Search the **in-memory** catalogue (name or code). No network call. |
| `GET`    | `/api/img?u=`                 | Server-side image proxy: fetches, downscales and re-serves a card image. |
| `GET`    | `/api/sync`                   | Sync status: catalogue card count, last sync time, in-progress flag, last error. |
| `POST`   | `/api/sync`                   | Start a background scrape of the official card list (~53 requests) and rewrite `catalogue.json`. Returns `202`; poll `GET /api/sync`. |
| `GET`    | `/api/cards/{id}`             | A catalogue card.                                           |
| `GET`    | `/api/stats`                  | Full stats scoped to the collection goal: completion, by family/owner/language/rarity. |
| `GET`    | `/api/sets`                   | Every set with completion (`owned`/`total` printed codes), in display order. |
| `GET`    | `/api/sets/{code}`            | All cards of a set (one per printed code) annotated with status (owned/ordered/wishlist). |
| `GET`    | `/api/collection?q=&owner=`   | The collection (from SQLite, offline). Filter by name / owner.|
| `POST`   | `/api/collection`             | Add a card. Body: `cardId`, `ownerId?`, `language`, `quantity`, `notes`, `status?` (`owned`/`ordered`/`wishlist`). |
| `POST`   | `/api/collection/batch`       | Add many cards in one transaction. Body: `{ items: [...] }`. |
| `PATCH`  | `/api/collection/bulk`        | Bulk-set the language of many copies. Body: `{ itemIds, language }`. |
| `PATCH`  | `/api/collection/{id}`        | Update owner/language/quantity/notes/status (qty ≤ 0 deletes an owned copy). Omitted fields are kept; `ownerId:null` clears the owner. |
| `DELETE` | `/api/collection/{id}`        | Remove a possession.                                         |
| `GET`    | `/api/collection/stats`       | Owned totals overall and per owner (header badge).           |
| `GET`    | `/api/owners`                 | List co-owners.                                              |
| `POST`   | `/api/owners`                 | Add a co-owner. Body: `{ "name": "…" }`.                     |
| `DELETE` | `/api/owners/{id}`            | Remove a co-owner (their cards become *Non attribué*, not deleted). |
| `GET`    | `/api/settings`               | Collection goal + included families.                         |
| `PUT`    | `/api/settings`               | Update the collection goal / families.                       |
| `GET`    | `/api/curated`                | List hand-added cards.                                       |
| `POST`   | `/api/curated`                | Add a card from a TCGplayer product URL/id. Body: `{ "url": "…" }`. |
| `DELETE` | `/api/curated/{id}`           | Remove a curated card (by its catalogue id).                 |

## MCP server

An [MCP](https://modelcontextprotocol.io) server is exposed at **`/mcp`** (streamable HTTP), so an assistant can query and manage the collection with the same store/catalogue as the REST API. Enabled by default (`MCP_ENABLED`); protect it with `MCP_TOKEN` if it's reachable beyond a trusted network (the tools include writes).

Tools:

- **Read** — `search_cards`, `get_card`, `list_sets`, `get_set`, `missing_cards`, `collection_stats`, `list_owners`.
- **Write** — `add_card` (owned/ordered/wishlist), `update_possession`, `remove_possession`, `set_language_bulk`, `add_owner`, `add_missing_card` (TCGplayer URL or manual entry).

Set `MCP_READ_ONLY=true` to expose only the read tools. Point an HTTP MCP client at `https://<host>/mcp` with header `Authorization: Bearer <MCP_TOKEN>`.

## Data model

**Catalogue** — `catalogue.json`: `{ syncedAt, count, cards: [...] }`, where each card is normalized (id, name, code, rarity, set, image) plus a `raw` object with the extra gameplay fields scraped from the official card list (color, type, power, counter, attribute, cost/life, effect, …). Loaded into memory at startup; the sync timestamp travels with the file.

**Collection** (SQLite, [`backend/schema.sql`](backend/schema.sql)):
- `owners` — co-owners (free-text names) managed in Préférences.
- `collection_items` — one row per **possession**: `(card_id, owner_id?, language, quantity, notes, status)`. `status` is `owned` / `ordered` / `wishlist`; ordered & wishlist rows drop language/quantity (they track a wanted card per owner). `card_id` references a card in `catalogue.json` (resolved at the app layer; absent cards still render as a stub).
  Adding the same `(card_id, owner_id, language, status)` again increments the quantity (owned only).
  Deleting an owner sets `owner_id` to NULL (`ON DELETE SET NULL`) so no card is lost.
- `curated_cards` — cards added by hand (TCGplayer-only promos the official list omits); merged into the in-memory catalogue at startup.

## Local development

Backend (needs Go 1.22+):

The backend reads a `.env` file automatically (looked up as `./.env` then
`../.env`, or via `ENV_FILE=`). **Run it from the repo root** so the local
defaults line up — `.env`, `data/` (the dev SQLite db) and `catalogue.json` all
sit at the repo root, next to each other:

```bash
(cd backend && go mod tidy)   # one-time
go run ./backend              # db -> ./data/, catalogue -> ./catalogue.json
```

Real environment variables still override `.env`, e.g. `PORT=9000 go run ./backend`.

Frontend (needs Node 20+), with hot reload and `/api` proxied to the backend:

```bash
cd frontend
npm install
npm run dev   # http://localhost:5173
```

`npm run build` type-checks and outputs the static SPA to `frontend/dist`, which the Go server serves in production.

## Notes & disclaimers

This project is not affiliated with Bandai, Eiichiro Oda, or Shueisha. Card data
and images are fetched from the official One Piece Card Game site and belong to
their respective owners; this app just helps you track what you own. **DON!!**
cards are absent from the official card list, so they're sourced from
[TCGplayer](https://www.tcgplayer.com/)'s public catalogue (data + images) and
merged into the catalogue at sync time — including the Premium Booster (PRB)
DON!! cards modelled as three editions (standard / foil / gold). Be polite
with sync (it runs ~53 requests against the official site) — run it occasionally,
not in a loop.
